## What this repo actually is

This is **not** a standalone Node app Рђћ it's a thin deployment wrapper that installs Ghost (https://ghost.org/) onto Heroku at build time. The Ghost runtime itself (`ghost.js`, `core/`, `versions/<ver>/`, etc.) does **not** live in git. It is materialized into the working directory during `npm install` by `bin/postinstall.sh`. `index.js` is a one-liner that `require('./ghost')` Рђћ that file only exists after postinstall runs.

Consequence: after a fresh clone, the repo will not run with `npm start` until postinstall has executed. Many "missing" files (`ghost.js`, `core/`, `LICENSE`, `PRIVACY.md`, `loggingrc.js`, `MigratorConfig.js`) are expected to be generated Рђћ `npm run clean` lists them.

## Deployment pipeline (Heroku)

The lifecycle is split across four distinct hook points. Editing any one requires understanding the others:

1. **Build Рђћ `npm install`** runs `bin/postinstall.sh`:
   - `ghost install <version> --dir /app/build Рђд --db sqlite3 --no-setup` installs Ghost into a throwaway build dir.
   - `package.json`, `package-lock.json`, and `node_modules` are copied **into** `/app/build/versions/*/` so Ghost's version dir has the user's full dependency tree merged in.
   - Then `/app/build/versions/*/*` is copied **back** over the repo root, bringing Ghost's generated files (including `ghost.js`) into place.
   - `bin/aws-s3.sh` copies the S3 storage adapter into `content/adapters/storage/s3/`.
   - `bin/themes.sh` copies each GitHub-sourced theme (`casper`, `headline`, `ease`, `source`) from `node_modules/` into `content/themes/`.
2. **Dyno boot Рђћ `.profile`** runs `bin/create-config`, which writes `config.production.json` from env vars (database URL, mail, S3, `APP_PUBLIC_URL`, `PORT`).
3. **Postdeploy Рђћ `app.json` Рєњ `bin/init-deployment`** waits for the DB (`bin/wait-for-db` polls MySQL until `SELECT 1` succeeds), then runs `node_modules/.bin/knex-migrator init --mgpath .` to initialize Ghost's schema.
4. **Run Рђћ `npm start`** = `node index.js` Рєњ `require('./ghost')`.

## Ghost version management

The Ghost version is **hardcoded in two places** that must stay in sync:
- `package.json` `"version"` field.
- The `ghost install <version>` argument in `bin/postinstall.sh`.

`.github/workflows/ghost-update-check.yml` runs weekly and opens a PR bumping both. A prior refactor tried reading the version from `package.json` inside the script; it was rolled back (see recent commits) Рђћ don't reintroduce that pattern without understanding why it failed (pnpm symlink / postinstall ordering issues).

`NPM_CONFIG_PRODUCTION=false` must be set on Heroku Рђћ otherwise devDependencies get pruned before postinstall finishes, breaking the Ghost install.

## Configuration surface

`bin/common/env-values.js` is the single source of truth for env-var aliases:
- **DB URL** resolves in order: `MYSQL_DATABASE_URL` Рєњ `JAWSDB_URL` Рєњ `CLEARDB_DATABASE_URL` Рєњ `JAWSDB_MARIA_URL`.
- **SMTP** resolves in order: `SMTP_LOGIN`/`SMTP_PASSWORD` Рєњ `MAILGUN_SMTP_LOGIN`/`MAILGUN_SMTP_PASSWORD`; service defaults to `Mailgun`.

`bin/create-config` parses the MySQL URL by hand (`url.parse`) rather than passing the connection string through Рђћ any change to the DB config shape has to go through `getMysqlConfig()`.

Storage: S3 is only enabled when `S3_ACCESS_KEY_ID` is set. Otherwise `fileStorage: false` and `storage: {}` Рђћ Ghost falls back to its default local storage (the README's "Cloudinary" claim is not reflected in the code; there is no Cloudinary adapter wired up).

## Common commands

- `npm start` Рђћ run Ghost (requires postinstall to have run).
- `npm run clean` Рђћ wipe everything generated by postinstall, including `node_modules`, `core/`, `components/`, installed themes in `content/themes/`, `.ghost-cli`, `ghost.js`, `LICENSE`, `loggingrc.js`, `MigratorConfig.js`, `PRIVACY.md`, and lockfiles. Use before testing a clean install.
- `npm run precommit` Рђћ lint-staged (runs `prettier-package-json --write` on `package.json`). Wired via husky.

There is **no test suite** and no lint/typecheck script in this repo. Don't fabricate one.

## When editing

- The repo has no TypeScript Рђћ ignore the global TypeScript guidelines for this project.
- Themes (`casper`, `headline`, `ease`, `source`) are pulled from GitHub via `package.json` deps and copied into `content/themes/` at postinstall. Don't edit files under `content/themes/<name>/` expecting them to persist Рђћ they're overwritten on every deploy. Edit the upstream theme repo or add a new one to `bin/themes.sh`.
- `content/` is Ghost's data directory (images, settings, sqlite DB in local dev). Treat as runtime state, not source.
- `versions/5.89.1/` is a stale leftover; the live version is whatever `bin/postinstall.sh` installs into `versions/<current>/` at build time.
